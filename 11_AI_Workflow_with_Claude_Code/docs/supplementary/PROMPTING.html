Prompting with RICE

A repeatable shape for prompts — so the quality of what Claude builds stops depending on how you happened to feel that morning.

Series Claude Code on a Greenfield Project
Read time ~12 minutes
Role Foundation for every scripted prompt

This guide introduces the prompting pattern used throughout the companion demo series. Every substantive prompt you'll see in the video scripts follows this structure. Read this once, and the scripted prompts become a lot easier to adapt to your own projects.

§ 01

Why a Framework at All?

Claude Code is only as good as the instructions it's given. In practice, that means the difference between a one-sentence request and a well-structured prompt is the difference between code you have to rewrite and code that ships.

The problem isn't that one-sentence prompts don't work — sometimes they do. The problem is that they're inconsistent. You can't reliably hand a prompt off to a teammate, paste it into a new session, or revisit it three weeks later without losing context. A framework gives your prompts a repeatable shape, so the quality doesn't depend on how you happened to feel that morning.

RICE is that framework.

§ 02

The RICE Framework

RICE stands for four elements that a good prompt almost always contains, whether you wrote them intentionally or not:

R
Role
Who is Claude acting as?
I
Input
What is the task?
C
Context
What constraints matter?
E
Execution
How should output be shaped?

The power of RICE is that it's a checklist. Before you send a prompt, scan it: does it have an R? An I? A C? An E? If any are missing, ask yourself whether that's intentional. Often it isn't, and filling the gap is where the prompt gets dramatically better.

2.1 · Role — who is Claude acting as?

The Role tells Claude what perspective, expertise, and priorities to bring to the task. A good Role does three things at once:

  1. Sets the expertise bar. "Senior backend engineer" signals different defaults than "junior developer learning Python."
  2. Names the priorities. "Security-conscious" vs. "performance-obsessed" vs. "readability-first" — these change the output meaningfully.
  3. Establishes the tone of reasoning. A "code reviewer" will flag issues; a "pair programmer" will suggest alternatives.
Weak
Write a function to hash passwords.
RICE Role
You are a senior backend engineer with a security-first mindset, working on a production FastAPI application that will store user credentials.

The second version implicitly tells Claude to pick bcrypt (not MD5), to use a proper salt, to validate the work factor, and to avoid logging the plaintext — without you having to spell any of that out.

2.2 · Input — what is the task?

The Input is the verb of the prompt: the concrete thing you want done. This is the part most people get right intuitively — "build X," "fix Y," "refactor Z." The trick is to keep it unambiguous and singular.

  1. One task per prompt. "Build the auth module and also set up the database and oh, while you're there, add logging" is three prompts pretending to be one.
  2. Name the artifact. Create backend/app/services/auth_service.py beats "handle auth."
  3. Say what success looks like. "…such that POST /auth/signup returns a JWT on success and 400 on validation failure" is a better Input than "…so users can sign up."
Weak
Add auth.
RICE Input
Create a signup and login flow. Add two routes — POST /auth/signup and POST /auth/login — each accepting {email, password} and returning a JWT on success.

2.3 · Context — what constraints, references, or background matter?

Context is the part most people underweight. It's everything Claude needs to know about the surrounding world: existing code, conventions, libraries, constraints, things to avoid, things to reuse. Without Context, Claude will invent conventions. With Context, it matches yours.

Include, when relevant:

  • Existing files or patterns to reference — "Follow the same structure as content_service.py."
  • Libraries and versions — "Use SQLAlchemy 2.x async style. We're on Pydantic v2."
  • What NOT to do — "Do not use synchronous database calls. Do not introduce a new ORM."
  • Standing rules that apply — "Follow the conventions in .claude/rules/backend-fastapi.md."
  • Upstream decisions — "We chose JWT over sessions because the frontend will be a separate deployment."
Weak
Add the signup route.
RICE Context
Add the signup route. Use SQLAlchemy 2.x async, Pydantic v2 for request/response models, and bcrypt for password hashing via passlib. Follow the same file structure as content_service.py. Do not log plaintext passwords, not even at DEBUG level.
!

Notice how much of that context is prohibitions. "Don't do X" is often more valuable than "do Y," because it rules out plausible-but-wrong paths Claude might otherwise take.

2.4 · Execution — how should the output be shaped?

Execution tells Claude what the deliverable looks like. Not what it does — what it is. This is where you specify format, structure, acceptance criteria, and (crucially) what you want Claude to do after the work is done.

Include, when relevant:

  • File paths to create or modify — "Create backend/app/routes/auth.py and update backend/app/main.py to register the router."
  • Output format — "Return the solution as a single code block, with no explanatory prose before or after."
  • Post-work actions — "After creating the files, run the tests with cd backend && uv run pytest tests/test_auth.py -v and report the output."
  • Verification criteria — "The route is working if curl to /auth/signup with a valid body returns a 200 with a JWT in the response body."
  • Commit behavior — "Stage and commit with message feat: add signup route with JWT response."
Weak
Also write tests.
RICE Execution
Create tests at backend/tests/test_auth.py. Cover: successful signup returns 200 + JWT; duplicate email returns 400; invalid email format returns 422. Run cd backend && uv run pytest tests/test_auth.py -v and report all failures. Do not commit until all tests pass.

§ 03

A Fully Worked Example

Here's the same task written three ways — watch how much Claude can work out on its own once the prompt is properly shaped.

Terrible — no framework
roll the dice
add login

This "works" only in the sense that Claude will do something. What it does depends entirely on what it happens to see first in the codebase and what defaults it has trained on. Good luck reproducing the result.

Decent — Input only
serviceable
Add a login route that takes email and password, verifies against the
users table, and returns a JWT on success.

Claude will probably do something reasonable. But it has no idea what the users table looks like, what password hashing scheme is already in use, or whether you want async or sync code.

Strong — full RICE
handoff-ready
**Role:** You are a senior backend engineer with a security-first mindset,
working on the ScriptSprout FastAPI application.

**Input:** Add a login route at `POST /auth/login` that accepts
`{email: str, password: str}` and returns `{access_token: str, token_type: "bearer"}`
on success.

**Context:**
- Follow the conventions in `.claude/rules/backend-fastapi.md`.
- Use SQLAlchemy 2.x async style. The `User` model already exists at
  `backend/app/models/user.py` with `email` (unique) and `password_hash` columns.
- Password hashes use bcrypt via `passlib.context.CryptContext` — an instance
  is already exported from `backend/app/services/auth_service.py` as `pwd_context`.
- JWT settings are in `backend/app/core/config.py` (`JWT_SECRET`, `JWT_ALGORITHM`,
  `JWT_EXPIRE_MINUTES`). Use the existing `create_access_token` helper from
  `backend/app/services/auth_service.py`.
- Do NOT log the plaintext password at any level.
- Do NOT return different error messages for "email not found" vs. "wrong password" —
  both should return 401 with the same generic body to prevent user enumeration.

**Execution:**
1. Create the route in `backend/app/routes/auth.py` (file already exists, add
   to it — do not replace).
2. Register nothing new in `main.py` — the router is already wired.
3. Add tests to `backend/tests/test_auth.py`. Cover: successful login returns
   200 + token; wrong password returns 401; unknown email returns 401 with the
   same body shape; malformed body returns 422.
4. After implementing, run `cd backend && uv run pytest tests/test_auth.py -v`
   and report the output. Do not stage or commit until all tests pass.
5. Once green, stage and commit with message: `feat: add login route with JWT response`.

That's a lot more text. It's also a prompt you could hand to a teammate, and they'd produce the same thing Claude does.

§ 04

When RICE is Overkill — The "As Appropriate" Clause

Not every prompt needs four labeled sections. Use RICE as appropriate:

  • Trivial operations — "stage and commit with message X," "run the tests and show me the output," "check git status" — stay as one-liners. Adding a Role block to "check git status" is cargo-culting.
  • Follow-up prompts within a single session — once Role and Context are established, subsequent prompts can reference them ("same role, same conventions — now add the logout route").
  • Quick diagnostic questions — "why is this test failing?" rarely needs a full framework.
If the prompt asks Claude to produce or modify meaningful code, use RICE. If it's a shell operation or a quick question, don't.

In the video scripts, you'll see this in practice. Major build prompts use the full R/I/C/E structure. Small operational prompts (create a branch, commit a file, push) don't — they're short enough to be unambiguous on their own.

§ 05

Common Failure Modes

A few patterns worth calling out, because they're the ones that trip people up most often:

  1. Role inflation. "You are the world's greatest engineer" adds nothing. Claude doesn't calibrate on superlatives. Calibrate on role and priorities instead: "senior backend engineer, security-first."
  2. Context overload. Dumping every file in the repo into the prompt doesn't help — Claude Code already reads the codebase. Context is for things Claude won't figure out on its own: conventions not in the code, prohibitions, upstream decisions, which of several plausible paths to take.
  3. Missing Execution. The most common gap. Without Execution, Claude will happily give you the code — but not run the tests, not commit the file, not tell you what to verify. You'll end up manually doing all the follow-up steps. Write Execution and reclaim that time.
  4. RICE as checklist theater. Four labeled sections with nothing meaningful in them is no better than no framework at all. If a section is truly empty, leave it out. If it's full of filler, cut it.
§ 06

How the Demo Uses RICE

Every major prompt in the demo scripts is structured as a RICE block. You'll see them written as: 

**Role:** ...
**Input:** ...
**Context:** ...
**Execution:** ...

Small prompts (commits, branch creation, quick diagnostics) stay as one-liners. This matches the "as appropriate" rule above.

6.1 · Role is stated once per phase, then inherited

Claude Code sessions have memory. Once you establish a Role in the first prompt of a phase, that Role carries forward through the rest of the session — you do not need to restate it on every prompt that follows.

So in the video scripts, you'll see the pattern:

  • First substantive prompt of each phase — full R/I/C/E block. Role is stated explicitly.
  • Every subsequent prompt in the same phase — I/C/E only. Role is inherited from the earlier prompt.
  • Exception — if the task genuinely shifts (e.g., "now act as a code reviewer and critique what we just built"), Role is restated for that prompt.

This is deliberate. Restating the same Role on every prompt is noise — Claude doesn't need it, and viewers don't benefit from seeing it repeated. What viewers should internalize is when to state Role, not that it always appears.

6.2 · Context evolves across the series

As you work through the demos, pay attention to how each prompt's Context changes over time. Early on, Context is mostly "here's the project, here are the rules files." By the middle of the series, Context often reads "you built this last phase — extend it, don't replace it." The framework doesn't change; what Claude needs to know does.

★

By the end of the demo series, you should be able to read an unfamiliar prompt, identify the four elements, and write your own for the next task you tackle. That's the goal.
